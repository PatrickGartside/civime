| Method | Path | Auth | Description |
|--------|------|------|-------------|
| GET | `/api/v1/health` | Public | Health check with DB status and metrics |

<!-- Auth values in table: Public, API Key, Token -->
Public = no auth required
API Key = X-API-Key header required (WordPress server-to-server calls)
Token = ?token={manage_token} query param required

<!-- Authentication section content -->
Two auth patterns:
  X-API-Key: WordPress sends this header on every server-to-server call.
    Never exposed to the browser. Set in WordPress wp-config.php / plugin settings.
    Required for: all POST/PATCH/PUT/DELETE public routes, all /admin/* routes.
  Token-based (manage_token): A 64-char hex string returned at subscription creation.
    Sent as ?token= query param. Scoped to a single user (covers all their subscriptions).
    Never expires. Used for: GET/PATCH/PUT/DELETE /subscriptions/{id} and the
    one-click unsubscribe link.
  confirm_token: A 64-char hex string used once to confirm opt-in.
    Sent via email link. Validated by GET /subscriptions/confirm or GET /reminders/confirm.
    Not cleared after use — confirmed_email flag is the authoritative state.

<!-- Subscription lifecycle walkthrough (for SUBSCRIPTION-LIFECYCLE.md) -->
All 7 steps with exact endpoints, token behavior, and DB state changes:
  Step 1: POST /api/v1/subscriptions (API Key)
    Body: { email, channels, council_ids, frequency, source }
    Response 201: { user_id, status: "pending_confirmation", manage_token, ... }
    DB: creates users row (confirm_token set), subscriptions rows (active=0)
    API: sends confirmation email with ?token={confirm_token}

  Step 2: [User clicks email link]
    GET /api/v1/subscriptions/confirm?token={confirm_token} (Public)
    DB: sets users.confirmed_email=1, subscriptions.active=1
    Response: 302 redirect to https://civi.me/notifications/confirmed

  Step 3: [User wants to manage preferences]
    GET /api/v1/subscriptions/{user_id}?token={manage_token} (Token)
    Returns: current preferences and subscription list

  Step 4: PATCH /api/v1/subscriptions/{user_id}?token={manage_token} (Token)
    Body: { email?, phone?, channels?, frequency? }
    Updates: user-level preferences

  Step 5: PUT /api/v1/subscriptions/{user_id}/councils?token={manage_token} (Token)
    Body: { council_ids: [...] }
    Effect: replaces full council subscription list (deactivates removed councils)

  Step 6: [One-click unsubscribe from email footer]
    GET /api/v1/subscriptions/unsubscribe?token={manage_token} (Public)
    DB: sets all subscriptions.active=0
    Response: 302 redirect to https://civi.me/notifications/unsubscribed

  Step 7: DELETE /api/v1/subscriptions/{user_id}?token={manage_token} (Token)
    Effect: same as unsubscribe — soft-deletes all subscriptions

Token behavior summary:
  confirm_token: one-time, sent by email, not cleared but superseded by confirmed_email=1
  manage_token: permanent, returned in 201 response, one per user, no expiry

    # Subscription Lifecycle

    Intro paragraph: This document extends the [DATA-FLOW.md sequence diagram](../architecture/DATA-FLOW.md)
    with API-level detail — exact endpoints, request/response shapes, and token behavior at each step.

    ## Overview

    Brief paragraph: A subscriber goes through: subscription creation → email confirmation →
    optional preference management → optional unsubscribe. Two tokens govern this flow:
    confirm_token (one-time opt-in) and manage_token (permanent self-service access).

    ## Token Model

    ### confirm_token
    - Location: users.confirm_token and reminders.confirm_token
    - Format: 64-character hex string
    - Generated: at subscription/reminder creation time
    - Use: ?token= param in confirmation email link
    - Lifecycle: not cleared after use — users.confirmed_email=1 is the authoritative state
    - Expiry: none — does not expire

    ### manage_token
    - Location: users.manage_token
    - Format: 64-character hex string
    - Generated: at subscription creation, returned in 201 response
    - Use: ?token= query param for all GET/PATCH/PUT/DELETE /subscriptions/{id} calls
    - Scope: one per user, covers all that user's subscriptions
    - Lifecycle: permanent — does not expire or rotate

    ## Lifecycle Steps

    Use numbered H3 sections (### Step 1: Create Subscription, etc.) for each of the 7 steps.
    For each step include:
    - The API call (method + path)
    - Auth requirement
    - Request body (if applicable)
    - Response summary
    - DB state change
    - Token role at this step

    Use code blocks for request/response examples where helpful.
    All 7 steps from the interfaces block must be documented.

    ## Reminder Flow

    Separate section for the meeting-day reminder flow (POST /reminders → GET /reminders/confirm):
    - POST /reminders: creates reminders row with reminders.confirm_token, sends email
    - GET /reminders/confirm?token={confirm_token}: sets reminders.confirmed=1, redirects
    - Note: uses reminders.confirm_token (separate from users.confirm_token)
